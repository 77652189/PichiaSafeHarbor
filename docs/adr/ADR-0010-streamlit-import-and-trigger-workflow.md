# ADR-0010：Streamlit 从只读展示改为可触发统一引擎的数据导入工作流

- 状态：Accepted
- 日期：2026-07-16
- 关系：修订 [ADR-0002](ADR-0002-streamlit-read-only-presentation.md) 中"网页只读，不负责下载数据、执行分析任务"的范围限制；ADR-0002 关于"页面不得复制科学规则、不得直接编辑结果"的核心约束继续有效，只是不再要求页面完全只读。

## 背景

ADR-0002 最初把 Streamlit 定位为纯只读展示层，原因是第一阶段想控制范围，避免网页和核心引擎出现两套逻辑。

但项目现在需要为"Strain-T 真实测序数据不定期到达"做好准备：如果每次新数据到来都必须先手动敲命令行完成校验、baseline、candidate-windows 等步骤，才能在网页上看到结果，会给不熟悉命令行的团队成员带来不必要的操作门槛。

真正需要防止的问题不是"网页能不能触发计算"，而是"网页会不会绕过或重新实现核心引擎逻辑"、"网页会不会绕过验收状态直接编辑结果"。这两者是可以和"网页可以触发运行"共存的。

## 决策

1. Streamlit 新增数据导入与运行触发能力：允许用户在网页上指向/上传新的参考数据（FASTA+GFF3 或等价参考 bundle），并触发 Slice 0 baseline、Slice 1 candidate-windows 等运行。
2. 触发运行时，网页只允许调用 `src/pichia_safe_harbor` 中已有的核心函数（如 `reference.py` 的校验函数、`pipeline.run_baseline`、`slice1.run_slice1`），不得在页面代码中重新实现校验、注释解析、区间计算或候选规则逻辑。核心引擎在代码里只有一份实现，命令行和网页调用同一份。
3. 每次网页触发的运行，产出格式、`run_id` 生成方式、三态状态模型（`execution_status`/`verification_status`/`scientific_acceptance_status`）、AcceptanceManifest 要求，与命令行触发的运行完全一致，没有网页专属的"弱化版"产物格式。
4. Run Catalog 的准入规则不变：只有通过 schema 校验、哈希核对、且达到相应科学验收状态的运行，才能进入正式候选浏览页面；执行失败或科学状态 `blocked` 的运行只能进入审计/诊断页面，网页必须清楚标注状态，不得把"刚触发完的运行"默认当成已验收。
5. 继续禁止：页面不得提供直接编辑某条候选的排除理由、证据等级或坐标的功能；不得在页面刷新/切换运行时静默复用旧运行的候选或缓存；不得让"网页触发"绕过任何 CLI 触发时同样适用的校验或停止线。要改变结果，只能是换输入数据或参数重新触发一次正式运行，不能手改产物。
6. 耗时较长的分析任务（例如未来 Slice 2 共线性核查，如果计算量大）应有明确的执行状态反馈（运行中/完成/失败），不得让网页假装同步完成而实际后台还在跑。

## 放弃的方案

### 继续保持完全只读，要求所有新数据都靠命令行处理

不利于团队里不熟悉命令行的成员使用，且会在真实数据到来时造成不必要的操作摩擦。

### 允许网页直接编辑已完成运行的候选/排除记录

会破坏"结果可追溯到输入、规则、参数"的核心不变量，一旦允许人工修改，审计链条就断了。

### 网页自己实现一套独立的候选计算逻辑图方便

会导致命令行和网页结果不一致，且需要双份维护/测试，长期成本更高。

## 后果

- `docs/ARCHITECTURE.md` 核心不变量、§4.9-4.12 Streamlit 相关章节需要修订，反映"可触发、不可复制规则、不可编辑结果"这套新平衡。
- `docs/REQUIREMENTS.md` §4.8 Streamlit 网页展示章节、§5 第一阶段不包含清单中与"只读"相关的条目需要相应调整。
- Slice 4（Streamlit）仍未被授权开始实现；本 ADR 只更新其架构设计，实际开发时间点仍由 `docs/EXECUTION_PLAN.md` 的门禁决定。
